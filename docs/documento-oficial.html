01

Visão Geral

Objetivo do Sistema

O Controle de Stages WMS é um sistema de Warehouse Management System (WMS) que controla a ocupação de posições em áreas de armazenagem (Stages). Cada Stage possui ruas e posições organizadas em grade. O sistema permite cadastrar paletes em posições específicas, pesquisá-los por TRO ou transportadora, e visualizar a ocupação em um mapa visual.

Problema que Resolve

Em armazéns e centros de distribuição, é fundamental saber exatamente qual palete ocupa qual posição, evitar duplicidade de ocupação, e localizar rapidamente um palete por seu identificador (TRO) ou transportadora. Sem um sistema dedicado, esse controle é feito manualmente (planilhas ou papel), sujeito a erros, perda de informação e dificuldade de pesquisa. O Controle de Stages WMS resolve isso com:

  • Mapa visual em tempo real da ocupação de cada stage.
  • Garantia de que cada posição abriga no máximo um palete (regra enforced no banco).
  • Pesquisa instantânea por TRO (normalizado) ou transportadora em todos os stages.
  • Persistência real em banco de dados — os dados sobrevivem a refresh, fechamento do navegador e redeploys.
  • Indicadores de ocupação (total, livres, ocupados, taxa) sempre atualizados.

Fluxo Operacional

Configuração inicial │ Cadastrar Stages (nome, ruas, posições) │ Cadastrar Transportadoras ▼ Uso diário │ Clicar em posição livre no mapa → cadastrar palete (transportadora + TRO + remessa) │ Clicar em posição ocupada → visualizar / editar / liberar │ Pesquisar por TRO ou transportadora → visualizar no mapa ▼ Gestão Excluir stages/transportadoras não utilizados Acompanhar indicadores de ocupação

Público-alvo

Operadores de armazém, supervisores de logística e equipes de expedição que precisam controlar a ocupação de posições em tempo real, com pesquisa rápida e visualização clara do estado do armazém.

Arquitetura

A arquitetura segue o princípio da Responsabilidade Única (SRP) em camadas bem definidas. Componentes nunca acessam o banco diretamente; regras de negócio vivem em hooks; lógica de pesquisa é pura; e a persistência é isolada em um único service.

Camada de Apresentação
components / pages — React + estilos inline
▼
Camada de Estado e Regras de Negócio
hooks — useWmsData, useToast, useMapHighlight, useDebouncedValue
▼
Camada de Serviços
services — storage.ts (persistência) · search.ts (pesquisa pura)
▼
Camada de Dados
Supabase / PostgreSQL — tables: stages, carriers, pallets

Comunicação entre Componentes

Os componentes se comunicam via props (dados de entrada e callbacks). O estado global da aplicação vive no hook useWmsData, instanciado em StagesPage. Modais e painéis recebem dados via props e notificam alterações via callbacks (onSave, onRelease, onViewOnMap).

Circulação de Dados

Na inicialização, useWmsData chama wmsStorage.loadAll() que consulta as três tabelas em paralelo e popula o estado local. A partir daí, toda escrita (criar/editar/liberar/excluir) atualiza o estado local e persiste no banco imediatamente. Derivados (occupiedByStage, indicators) são memoizados e recalculam apenas quando a fonte muda.

Como funciona o Banco

O Supabase expõe um endpoint REST sobre PostgreSQL. O cliente usa a anon key (segura para operações permitidas pelo RLS). As tabelas stages, carriers e pallets têm Row Level Security habilitado com políticas que permitem CRUD completo para anon e authenticated (single-tenant, sem autenticação por enquanto). Uma constraint UNIQUE(stage_id, street, position) garante no nível do banco que duas posições não sejam ocupadas pelo mesmo palete.

Como funciona o Mapa

O StageMap gera a grade dinamicamente a partir da configuração do Stage: colunas = ruas, linhas = posições (sempre de cima para baixo). Cada célula é um botão circular colorido: verde (livre) ou vermelho (ocupado, com ícone de pacote). Posições destacadas pela pesquisa recebem borda azul e glow. O componente é memoizado com React.memo para evitar re-renderizações quando apenas outros stages mudam.

Como funciona a Pesquisa

O SearchPanel usa useDebouncedValue (350ms) para evitar consultar a cada tecla. A lógica pura em search.ts normaliza o TRO (remove prefixo, espaços, zeros à esquerda) para correspondência exata, ou filtra por parte do nome da transportadora (case-insensitive). A pesquisa sempre considera todos os stages. Resultados são ordenados por Stage → Rua → Posição e agrupados por Stage.

Constraint Única

UNIQUE (stage_id, street, position) — garante a regra de negócio RN-01 no nível do banco: é impossível que dois paletes ocupem a mesma posição, mesmo que a validação no cliente falhe.

Índices

ÍndiceTabela/ColunaFinalidade
IDX idx_pallets_stage_idpallets.stage_idAcelera filtro por Stage
IDX idx_pallets_carrier_idpallets.carrier_idAcelera filtro por transportadora
IDX idx_pallets_tropallets.troAcelera pesquisa por TRO

Trigger

update_pallets_updated_at — atualiza automaticamente a coluna updated_at em qualquer UPDATE da tabela pallets, via função PL/pgSQL update_updated_at_column().

RLS (Row Level Security)

Todas as tabelas têm RLS habilitado. Políticas permitem CRUD completo para anon e authenticated (single-tenant, sem autenticação por enquanto). Quando autenticação for adicionada, basta trocar USING (true) por USING (auth.uid() = user_id).

Fluxo de Gravação

Componente → useWmsData (valida regra de negócio) → setEstado local (atualização otimista) → wmsStorage.saveXxx() → Supabase (upsert/delete via REST) → PostgreSQL

Fluxo de Leitura

useWmsData (useEffect na inicialização) → wmsStorage.loadAll() → Supabase (SELECT em paralelo: stages, carriers, pallets) → mapeia rows snake_case → camelCase → setEstado local → componentes renderizam

Combobox pesquisável para seleção de transportadora. Filtra por nome, fecha ao clicar fora, mostra check no item selecionado. Garante que a transportadora seja sempre selecionada da lista (RN-02).

08

Hooks

useWmsData

Hook central de dados. Concentra TODAS as regras de negócio de leitura/escrita. Carrega dados do Supabase na inicialização. Toda escrita atualiza o estado local e persiste no banco imediatamente.

Expõe:

  • Estado: loading, stages, carriers, pallets, lastSync, occupiedByStage, indicators.
  • Operações: createStage, deleteStage, createCarrier, deleteCarrier, savePallet, releasePallet, refresh.
useToast

Gerencia notificações temporárias. showToast(message, type) exibe uma toast que desaparece após 2800ms. Cancela toast anterior se uma nova for exibida.

useMapHighlight

Gerencia o destaque de posições no mapa. highlightOnStage(stageId, positionKeys) destaca posições em azul e remove o destaque após 3200ms. Inclui limpeza do timer no desmontar.

useScrollToHighlight

Centraliza a primeira posição destacada na tela via scrollIntoView.

useDebouncedValue

Atrasa a propagação de um valor em delay ms. Usado pela pesquisa para evitar consultar a cada tecla (350ms).

09

Services

services/storage.ts

Camada de persistência usando Supabase. Isola toda a lógica de acesso ao banco. Mapeia campos snake_case (DB) para camelCase (TS).

Funções:

  • loadAll() — carrega stages, carriers e pallets em paralelo (Promise.all).
  • saveStages(stages) — sincroniza a lista de stages (upsert + delete diff).
  • saveCarriers(carriers) — sincroniza a lista de carriers.
  • savePallets(pallets) — sincroniza a lista de pallets.
services/search.ts

Lógica de pesquisa pura (sem estado, sem side-effects).

Funções:

  • searchPallets({ type, query, pallets, stages, carriers }) — pesquisa por TRO (correspondência exata após normalização) ou transportadora (parte do nome, case-insensitive). Sempre pesquisa em todos os Stages. Retorna resultados ordenados por Stage → Rua → Posição.
  • groupResultsByStage(results) — agrupa resultados por Stage, preservando a ordem.

Fluxos Operacionais

Cadastrar Palete

1. Usuário clica em posição livre no mapa 2. StagesPage abre PositionModal (modo cadastro) 3. Usuário seleciona transportadora (combobox), digita TRO, remessa opcional 4. Usuário clica "Salvar" 5. PositionModal valida campos obrigatórios 6. StagesPage.handleSavePallet → useWmsData.savePallet 7. savePallet verifica conflito de posição (RN-01) 8. Cria registro, atualiza estado local, persiste no Supabase 9. Em sucesso: fecha modal automaticamente (RN-13) 10. Toast de sucesso exibido

Editar Palete

1. Usuário clica em posição ocupada no mapa 2. PositionModal abre em modo visualização 3. Usuário clica "Editar" → modal muda para modo edição 4. Usuário altera transportadora/TRO/remessa 5. Clica "Salvar" 6. savePallet (modo edição) atualiza o registro no estado e no banco 7. Trigger update_pallets_updated_at atualiza updated_at 8. Modal fecha, Toast exibido

Liberar Posição

1. Usuário clica em posição ocupada 2. PositionModal abre em modo visualização 3. Usuário clica "Liberar posição" 4. useWmsData.releasePallet remove o palete do estado 5. wmsStorage.savePallets persiste (delete diff) 6. Modal fecha, Toast de sucesso exibido 7. Posição volta a ficar verde (livre) no mapa

Pesquisar

1. Usuário seleciona tipo (TRO ou Transportadora) no SearchPanel 2. Digita query 3. useDebouncedValue aguarda 350ms (RN-12) 4. searchPallets normaliza query e filtra pallets em todos os Stages 5. Resultados ordenados por Stage → Rua → Posição (RN-09) 6. groupResultsByStage agrupa por Stage 7. Grupos renderizados com botão "Visualizar no mapa"

Visualizar no Mapa

1. Usuário clica "Visualizar no mapa" em um grupo de resultados 2. StagesPage.handleViewOnMap troca o Stage ativo 3. useMapHighlight.highlightOnStage destaca as posições em azul 4. useScrollToHighlight rola a tela para a primeira posição 5. Destaque remove-se automaticamente após 3,2s (RN-11)

Como Configurar o Ambiente

O arquivo .env na raiz do projeto deve conter as variáveis do Supabase: 

VITE_SUPABASE_URL=https://<projeto>.supabase.co
VITE_SUPABASE_ANON_KEY=<chave-anon>

Essas variáveis são lidas em src/services/storage.ts via import.meta.env. Os tipos estão declarados em src/vite-env.d.ts.

Como Configurar o Banco

A migration supabase/migrations/001_create_wms_tables.sql cria todas as tabelas, índices, constraints, trigger e políticas RLS. Aplicar via Supabase MCP ou painel do Supabase. Após aplicada, o sistema está pronto para uso — sem dados mock, sem LocalStorage, sem SessionStorage.

Resumo Executivo

O projeto é uma Single Page Application (SPA) escrita em TypeScript sobre React 18, com build via Vite e persistência em PostgreSQL gerenciado pelo Supabase. O gerenciador de pacotes é npm. Não utiliza TailwindCSS — a estilização é feita via design tokens em TypeScript aplicados como estilos inline, garantindo um único ponto de verdade visual.
